# Pacific Conflict — Data Tools

Scripts to inspect the **Pacific Conflict** binary (255,360 bytes) and derive section layout **without** using DAT_ symbols from the decompiler. Output is used to define import ranges for Ghidra and to document the on-disk format.

## Input file

Default path (relative to `tools/`):

```
../../../Hellcats/Pacific Conflict
```

Override with the first positional argument to either script.

## 1. inspect_pacific_conflict.py

**Purpose:** Parse header (u32/u16), detect section tables, scan for code-like regions (68k opcode density), output JSON.

**Run:**

```bash
cd tools
python3 inspect_pacific_conflict.py [path_to_binary]
```

**Options:**

- `-o FILE` — Write full report JSON to `FILE`.
- `--sections-json FILE` — Write only the sections list (and file_size) to `FILE` (e.g. `../sections.json`).

**Output (stdout or `-o`):**

- `header_u32` / `header_u32_dec` — First 4 u32 (big-endian).
- `header_u16_first_16` — First 16 u16.
- `table_candidates` — If any header value is treated as the start of a u32 table: entry count, monotonic, first/last.
- `code_like_region_count` / `code_like_regions_sample` — Windows (64 B, stride 32) with high 68k signature density.
- `sections` — Candidate sections with `offset`, `size`, `entropy`, `code_score`, `source`.

**Interpretation:**

- **code_score** — Fraction of words in the first 1K that match known 68k opcodes (RTS, LINK, LEA, MOVEA, etc.). Higher = more likely code.
- **entropy** — Byte entropy over the section (capped 64K). High entropy + low code_score = data or mixed.
- **table_candidates** — If `monotonic` is true and entries are in file range, the header value may be a section directory (list of offsets).

## 2. scan_code_windows.py

**Purpose:** Score fixed-size windows by 68k opcode pattern density; output top offsets.

**Run:**

```bash
python3 scan_code_windows.py [path_to_binary]
```

**Options:**

- `-w SIZE` — Window size in bytes (default 256).
- `-s STRIDE` — Stride in bytes (default 16).
- `-n N` — Number of top offsets to print (default 80).
- `--skip-header N` — Skip first N bytes (default 0x20).
- `-o FILE` — Write JSON (`top_offsets`, `window`, `stride`) to `FILE`.

**Interpretation:**

- Top-scoring offsets are the best candidates for **code** in the file.
- Use with `sections.json`: if a section has high `code_score` and contains these offsets, treat it as code for Ghidra import.

## 3. sections.json

Generated by `inspect_pacific_conflict.py --sections-json ../sections.json` (or equivalent). Contains:

- `file_size` — 255360.
- `sections` — List of `{ "name", "offset", "size", "entropy", "code_score", "source" }`, ordered by `offset`.

**Recommended use:**

- **Import as code in Ghidra:** Prefer sections with `code_score` ≥ ~0.05–0.08 (e.g. `segment_1_after_header`, `at_offset_u32[2]`, `at_offset_u32[3]`).
- **Import as data or skip:** Sections with very low `code_score` (e.g. 0.0078) and lower entropy may be data or BSS-like; import as data or leave for manual analysis.

## 4. Section directory

The inspector checks whether header u32 values (0x107, 0x1a75c, 0xdc8, 0xb2a4) point **into** the file and whether the bytes there form a **table** of u32s:

- **In-range:** Each u32 in the table is in [0x20, file_size).
- **Monotonic:** Offsets increase (or stay equal).
- **Plausible size:** Segment lengths implied by consecutive offsets are at least a few bytes.

If `table_candidates` contains an entry with `monotonic: true` and several entries, that header field may be a section directory. The current file does not show a clear directory table at those offsets; the header is instead interpreted as segment lengths / segment start offsets (see `Pacific_Conflict_loader_analysis.md`).
